RL Kill Switch

A ROS2 safety node that guards an autonomous tugboat driven by a reinforcement learning policy. The kill switch sits between the RL inference node and the Arduino motor controller, continuously monitoring six independent failure conditions and overriding all propulsion commands the moment any condition is violated.

What It Does

The tugboat's RL policy runs at 10 Hz and publishes raw PWM commands to /tug1/PWM_raw. Under normal conditions the kill switch acts as a transparent relay, forwarding those values unchanged to /tug1/PWM and to the Arduino over serial.

When any of six health triggers fires, the node immediately substitutes [0.0, 0.0] — neutral thrust and servo — regardless of what the policy outputs. The Arduino receives the same neutral command over the serial link, bringing the vessel to a stop. The RL policy keeps running, but its commands are suppressed until the system is declared safe again.

Six Safety Triggers

Each trigger operates independently. Any single trigger activates the kill output immediately.

ID Condition Detection method Recovery
T1 Vessel outside permitted boundary GPS fix converted to tank coordinates, then ray-casting polygon test on each GPS callback Manual reset
T2 RC link or serial connection lost Arduino sends RC:1 / RC:0 heartbeat every 100 ms; timeout after 1 s without RC:1 Manual reset
T3 Operator PC unreachable Background thread sends ICMP ping every 1 s; triggers if RTT exceeds 500 ms or host is down Auto recovery
T4 GPS rate degraded FrequencyMonitor: fires after 3 consecutive 1-second windows below 3 Hz Auto recovery
T5 PWM values outside safe range Inline range check on every /tug1/PWM_raw message (thruster: 1500–1730, servo: 1100–1900) Manual reset
T6 IMU rate degraded FrequencyMonitor: fires after 3 consecutive windows below 30 Hz Auto recovery

Triggers T3, T4, and T6 (remote comms and sensor rate) clear automatically when the condition resolves. Triggers T1, T2, and T5 (boundary, RC link, and PWM range) require a manual service call after the root cause is confirmed resolved.

State Machine

NORMAL

All triggers clear. PWM commands forwarded as-is to the Arduino.

TRIGGERED

One or more auto-recovery triggers active. Kill output active. Clears automatically when triggers resolve.

ARMED

A manual-reset trigger (T1, T2, or T5) fired. Kill output held until the /kill_switch/reset service is called.

The ARMED state is intentionally sticky: even if the trigger condition clears on its own, the kill remains active until an operator explicitly confirms it is safe to resume. The reset service verifies that no manual-reset trigger is still active before accepting the call.

Frequency Monitor

Sensor rate degradation (T4, T6) is detected by FrequencyMonitor, a standalone utility class that uses a sliding time window rather than a fixed sampling interval.

On each sensor callback, the monitor appends a timestamp and evicts entries older than window_sec (1 second). The current rate is count / window_sec. If this falls below min_hz, a fail counter increments; otherwise it resets to zero. Only after consecutive_fail (3) windows in a row does the monitor return True — preventing transient bursts from causing false alarms.

GPS thresholds

min 3 Hz, 3 consecutive windows → ~3 seconds to trigger

IMU thresholds

min 30 Hz, 3 consecutive windows → ~3 seconds to trigger

Design Decisions

Serial ownership moved to the kill switch. The original inference node held the Arduino serial connection and sent PWM directly. Relocating serial I/O to the kill switch ensures that a crashed or misbehaving inference node cannot issue motor commands — the Arduino only receives output from the safety layer.

Minimal change to the inference node. Only two lines in tugboat_inference_node.py needed to change: renaming the output topic from /tug1/PWM to /tug1/PWM_raw, and removing the serial initialization and send timer. All policy logic, observation assembly, and coordinate math remain untouched.

No dependency on the ROS2 graph for safety-critical checks. T2 (RC link) and T3 (operator PC reachability) are detected by background threads that communicate directly over serial and ICMP, respectively. A failure in the ROS2 middleware or DDS layer does not prevent these triggers from firing.

Lock discipline over all shared state. Trigger flags, state machine state, and the last RC heartbeat timestamp are mutated only while holding threading.Lock. ROS callbacks, background threads, and the 50 Hz timer all contend on the same lock, preventing races between concurrent writers.

Configurable entirely through YAML. Every threshold — boundary polygon vertices, sensor minimum rates, ping latency limit, PWM safe ranges, RC timeout — is exposed as a ROS2 parameter and loaded from kill_switch_params.yaml at launch. No recompilation is needed to adapt the node to a new deployment site.

Deployment

Before first deployment, update three parameters in config/kill_switch_params.yaml:

ParameterWhat to set
main_pc_ip IP address of the operator's PC on the field network
boundary_gps_points GPS survey of the actual tank boundary polygon (flat lat/lon list)
serial_port Device path for the Arduino (default: /dev/ttyACM1)
cd ~/ros2_ws
colcon build --packages-select kill_switch
source install/setup.bash
ros2 launch kill_switch kill_switch.launch.py

# monitor status
ros2 topic echo /tug1/kill_switch/status

# manual reset after resolving T1 / T2 / T5
ros2 service call /tug1/kill_switch/reset std_srvs/srv/Trigger {}
